Distributed Tensorflow, Concept and Overview

Overview

Distributed tensorflow has been released more than a year and a half now and is now widely adopted in neural network training. This post helps me, and hopefully, you, to understand various concepts at differenr level better, such as worker, master, replica, clone, chief, job, task, etc. A nice overview video can be found here

In-graph Replication

In-graph replication provides minimal but still beneficial parallization in training. It simply split up input in to equally sized chunk, enabling multiple devices (video cards) to work together. Two modes are available: synchrounous, where parameter server would wait and gather gradients and wait for all workers to finish before proceeding to the next batch and asynchronous which is of opposite.

In-graph replication is relatively easy to achieve. We simply iterating through devices when constructing inputs. When training on a single machine with multiple video cards, no or little network bottleneck exists and this is the technology we would like to adopt.

Between-graph Replication

One concept to be cleared before we go through between-graph replication. A client refers to a trainig process. The most typical example would be our python script constructing the graph and most importantly, run a ‘session’. Session is the bridge higher level APIs ‘talks’ to kernels and ops in Tensorflow.

When using between-graph replication, multiple client run together. The way they synchronize on model parameter values is by sharing a parameter server. And any two clients that create a variable with the same name on the same device will share the same backing storage for that variable.

Sometimes, when the size of paramter pool grows so large that, it won’t fit into one device. Tensorflow provides several algorithm to place variables into different devices, including Robinhood and greedy. The main concern is that some paramter entity (such as W) can be siginificantly larger than others (such as b). Imbalance replacement may lead to traffic imbalance and performance degradation.

What if a parameter entity is too large to fit? Partitioning it is still possible with tensorflow.

Parameter Server Notes

In DisBelief, parameter server and worker replicas run different program, programmed in diffenrent APIs. In Tensorflow, ps job and worker job run the same code. GPU can even be utilized to accelerate parameter update, as an evidence of the flexibility it offers. And workers are able to cache state.

Parameter server is a fragment of data flow graph and is not implemented in the low level excution engine. It accept data flow and gradient instruction from workers. So in a ps job/client, the process only ‘join’.

Concept Clarification

  • Job: Logical set of resources dedicated for one purpose. For example, worker job and ps job.

  • Task: Subset of resources from job, facilitate resource partiotioning and replication.

  • Replica: A copy of graph. Oftenly a single machine.

  • Clone: Used in object detection API, refering to number of tasks.

  • Chief: A special worker responsible for logging, checkpointing, etc.
